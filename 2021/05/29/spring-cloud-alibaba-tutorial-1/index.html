本文记录了Spring Boot 集成 Apache Dubbo 的过程。参考书目：Spring Cloud Alibaba 微服务原理与实战。

初步集成Dubbo

假设我们想远程调用的服务为类HelloService中的sayHello(name)方法，如果这个方法仅仅是一个普通的方法：

那么它想要有对外提供服务的能力起码要有一个寻址方法能够调用这个方法，一个简单的思路是写一个Controller，在这个Controller中接收来自外部的请求并产生调用行为，这个包装类很像serverless中的handler类。在这里，这个类叫Provider。它理应将HelloService这个平庸的类做成服务，对外发布，并能调用。显然，还应该有个类叫Consumer，这个Consumer应该可以和Provider不必生存在同一进程，却能通讯，与浏览器中js脚本通过http协议远程调用后台接口获取资源渲染页面一般。既然如此，这两个类似乎不必相互依赖，甚至不必同种语言，只要它们同意采用一个协议，亦即代码的编写方式从类库转变为服务。此处，Dubbo 解决了两个RPC中的问题：如何表示数据，如何传递数据。另一方面，它还应该能解决另一个问题——如何确定方法。到这里我还没有读过 dubbo 的源码，如果让我妄下断言，我可能会在Consumer这个类中采用类似@Resource注解的方式，将远程暴露的类实例化为一个bean，如果这样，这要求我们的技术栈能够远程传输对象。为了简便，我们还可以这样做，让两个类都采用同种语言编写，将HelloService做成接口，在Provider中实现它，在Consumer中调用，这样就实现了二者的解耦。当然，这个方案，仅仅是我看到Provider和Consumer这两个类，以我仅有的对RPC肤浅的认识思考到的简单做法。

集成 ZooKeeper

mac 下安装 ZooKeeper：brew install zookeeper。

之前简单的实践中，我们默认消费者类知道了服务的地址，并且这里服务与消费是单对单的。如果我们考虑将服务做成多个节点放在集群中，每个节点都有一个地址，那么接踵而来的问题就是——消费者该选择那个接口调用？试想，如果客户端（消费者）维护一个地址表，有一个线程来嗅探已保存的服务节点，如果节点死亡，就把它删除; 但是它无法添加新增的节点，为此它还需要能从一个池子中捞取新节点，而这个池子应该是它与服务提供者共同约定好的; 此外，它还应该能监控这些节点的性能，以便能实现负载均衡。那么显然，我们需要一个第三者——服务分发者来实现，它应该能至少实现服务的注册与发现，负载均衡。在这里zookeeper就可以实现这些功能。我记得我之前写过 docker 相关的客户端，利用 redis 来记录实例化的 docker 容器，以函数的id为键，并且利用一个线程轮询获取容器的状态。由于初生牛犊不怕虎，不知道这里的水深，倒是写了下去，不过可想而知它的性能，这里也算是实现了服务的注册和发现了吧哈哈。总之，这里我们需要一个第三方参与。

根据书中描述：

当 Dubbo服务消费者启动时,会对/dubbo/com.akira.api.IHelloService/providers节点下的子节点注册Watcher监听，这样便可以感知到服务提供方节点的上下线变化,从而防止请求发送到已经下线的服务器造成访问失败。同时,服务消费者会在/dubbo/com.akira.api.IHelloService/consumers下写入自己的URL ,这样做的目的是可以在监控平台上看到某个Dubbo服务正在被哪些服务调用。最重要的是, Dubbo服务的消费者如要调用IHdloService服务,那么它会先去路径/dubbo/com.akira.api.IHelloService/providers下获得所有该服务的提供方URL列表,然后通过负载均衡算法计算出一个地址进行 远程访问。

显然按此描述，服务提供者对消费者并不是透明的。我理想中的服务注册，发现与治理理应不涉及到消费者，由服务分发平台监控性能，选择服务调用哪个节点，就像一个供应商，顾客虽然看不到价格，但是由供应商去选择最便宜的出售给顾客，这样就能保障顾客永远购买的是最便宜的商品。因此，如果仅仅是利用 zookeeper 的临时节点，持久化节点，Watcher 机制这些其他平台也可以实现的特性来实现的“第三方资源池”这样的服务注册与发现机制，我觉得我不会太愿意去使用它。
