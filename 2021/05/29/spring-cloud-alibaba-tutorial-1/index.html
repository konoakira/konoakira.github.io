初步集成Dubbo

假设我们想远程调用的服务为类HelloService中的sayHello(name)方法，如果这个方法仅仅是一个普通的方法：

那么它想要有对外提供服务的能力起码要有一个寻址方法能够调用这个方法，一个简单的思路是写一个Controller，在这个Controller中接收来自外部的请求并产生调用行为，这个包装类很像serverless中的handler类。在这里，这个类叫Provider。它理应将HelloService这个平庸的类做成服务，对外发布，并能调用。显然，还应该有个类叫Consumer，这个Consumer应该可以和Provider不必生存在同一进程，却能通讯，与浏览器中js脚本通过http协议远程调用后台接口获取资源渲染页面一般。既然如此，这两个类似乎不必相互依赖，甚至不必同种语言，只要它们同意采用一个协议，亦即代码的编写方式从类库转变为服务。此处，Dubbo 解决了两个RPC中的问题：如何表示数据，如何传递数据。另一方面，它还应该能解决另一个问题——如何确定方法。到这里我还没有读过 dubbo 的源码，如果让我妄下断言，我可能会在Consumer这个类中采用类似@Resource注解的方式，将远程暴露的类实例化为一个bean，如果这样，这要求我们的技术栈能够远程传输对象。为了简便，我们还可以这样做，让两个类都采用同种语言编写，将HelloService做成接口，在Provider中实现它，在Consumer中调用，这样就实现了二者的解耦。当然，这个方案，仅仅是我看到Provider和Consumer这两个类，以我仅有的对RPC肤浅的认识思考到的简单做法。

集成 ZooKeeper

mac 下安装 ZooKeeper：brew install zookeeper。
