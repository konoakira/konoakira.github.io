既然不同容器可以使用不同的 Network Namespace，那外界又是如何与这个容器通信呢，或者容器之间又是如何通信的呢？当然，我们可以使用docker run -d -net=host ...让容器使用主机的网络。这样直接的方式可以为容器提供良好的网络性能，但也会不可避免地引入共享网络资源的问题，比如端口冲突。所以，在大多数情况下，我们都希望容器进程能使用自己Network Namespace里的网络栈，即：拥有属于自己的IP地址和端口。

Veth Pair

任何一个计算机领域的问题都可以通过增加一个中间层解决，要想让两个容器网络连接，我们一个直接的想法就是让它们共同连接到一个交换机上（交换机内部的CPU会在每个端口成功连接时，通过将MAC地址和端口对应，形成一張MAC表。在今后的通讯中，发往该MAC地址的数据包将仅送往其对应的端口，而不是所有的端口）。Docker项目会默认在宿主机上创建一个名叫docker0的网桥，凡是连接在docker0网桥上的容器，就可以通过它来进行通信，接下来我们需要让容器连接到交换机上了。而Veth Pair就起到了连接的作用，它被创建出来后，总是以两张虚拟网卡（Veth Peer）的形式成对出现的。并且，从其中一个“网卡”发出的数据包，可以直接出现在与它对应的另一张“网卡”上，哪怕这两个“网卡”在不同的Network Namespace里。

跨主通信

如果两台不同的宿主机上都有 docker 容器，它们想互相通信，我们似乎也可以像上面这么做，尽管两台宿主机的docker 相互独立。我们可以利用Overlay Network技术，在已有的宿主机网络上，再通过软件构建一个覆盖在已有宿主机网络之上的、可以把所有容器连通在一起的虚拟网络；或者，通过配置宿主机的路由表，数据包转发到正确的宿主机上。

Flannel

Flannel是一种解决跨主通信的方案，它支持三种后端实现，分别是：

  1. VXLAN
  2. host-gw
  3. UDP

UDP

UDP是最简单也是性能最差的一种实现，如今已被弃用。跨主通信要解决的核心问题就是宿主机 A 上容器发出的请求如何到达宿主机 B 上的容器。宿主机 A 上的docker0 显然不会知道宿主机 B 上的 docker0 的ip（为方便叙述，叫ip_b)的。纯粹直觉的一个想法是映射，宿主机 A 上请求到 ip_b 网段，统一映射发往 B 的ip上的某一个端口，同时在 B 上监听这一端口，将其接收的数据发往 B 上的 docker0。要实现这一功能，我们至少需要

  1. 拦截，转发 docker0 网卡出来（源地址，目标地址为docker0 ip）的数据
  2. 能够分布式地管理这些映射，达成分布式共识
  3. 能够实现 docker0 的注册以及发现

Flannel就实现了这些功能。所有的 docker0 的ip在其下称为子网（subnet），存储在 etcd中作为key, (etcd 与 zookeeper 很像，分布式存储，这样每台宿主机能够达成共识，这也意味着它们的 docker0 ip 不同)，那么 value显然就是宿主机的 ip （或者套接字）了。每次，我们在宿主机上新搭建一个 docker 时由 flannel 负责分布 subnet ip，写入 etcd，实现注册。同时，flannel 还会更新宿主机的路由表，将发往其他宿主机docker0的请求拦截封装成UDP转发到该宿主机 ip 的8285端口。另一端的flanneld进程监听这个端口，将UDP请求解封转发给 docker0。

这个过程（转发）相当耗费性能：

  • 用户态的容器进程发出的IP包经过docker0网桥进入内核态

  • IP包根据路由表进入TUN（flannel0）设备，从而回到用户态的flanneld进程

  • flanneld进行UDP封包之后重新进入内核态，将UDP包通过宿主机的eth0发出去

  • UDP 封装和解封在用户态完成

VXLAN

VXLAN，即Virtual Extensible LAN（虚拟可扩展局域网），是Linux内核本身就支持的一种网络虚似化技术。所以说，VXLAN可以完全在内核态实现上述封装和解封装的工作，从而通过与前面相似的“隧道”机制，构建出覆盖网络（Overlay Network）。

VXLAN，即Virtual Extensible LAN（虚拟可扩展局域网），是Linux内核本身就支持的一种网络虚似化技术。所以说，VXLAN可以完全在内核态实现上述封装和解封装的工作，从而通过与前面相似的“隧道”机制，构建出覆盖网络（Overlay Network）。

VXLAN的覆盖网络的设计思想是：在现有的三层网络之上，“覆盖”一层虚拟的、由内核VXLAN模块负责维护的二层网络，使得连接在这个VXLAN二层网络上的“主机”（虚拟机或者容器都可以）之间，可以像在同一个局域网（LAN）里那样自由通信。当然，实际上，这些“主机”可能分布在不同的宿主机上，甚至是分布在不同的物理机房里。而为了能够在二层网络上打通“隧道”，VXLAN会在宿主机上设置一个特殊的网络设备作为“隧道”的两端。这个设备就叫作VTEP，即：VXLAN Tunnel End Point（虚拟隧道端点）。这里VTEP设备的作用，其实跟前面的flanneld进程非常相似。只不过，它进行封装和解封装的对象，是二层数据帧（Ethernet frame）；而且这个工作的执行流程，全部是在内核里完成的（因为VXLAN本身就是Linux内核中的一个模块）。

docker0出来的数据还是要经过flannel.1设备发出，它也是源VTEP设备，那么它怎么知道目的VTEP设备的地址呢？（在UDP模式中，我们需要知道flanneld进程的端口号），这回，我们需要知道（假设主机在同一个局域网下）目的VTEP设备的mac地址。这个地址显然可以在 docker0 启动时注册叫 etcd 中。

VXLAN利用了linux的特性，可以在内核态中处理数据。与UDP模式一样，我们找到一个容器所需要的信息并不会减少。

Linux 内核要负责将数据转发到 docker0，这回它将我们的数据包封装成内部数据帧（Inner Ethernet Frame），帧的首位放目的VTEP设备的mac地址，第二位放目的容器ip地址，这样就（当数据包到达目的主机）就实现了转发功能。数据帧的后面放从 docker0 出来的原始数据报文。至于想要到达目的主机，Linux内核还需要再把内部数据帧进一步封装成为宿主机网络里的一个普通的数据帧，好让它“载着”“内部数据帧”，通过宿主机的eth0网卡进行传输。我们把这次要封装出来的、宿主机对应的数据帧称为“外部数据帧”（Outer Ethernet Frame）。

现在数据可以到达目的主机，也可以转发到docker0，但我们还需要一个标识告诉Linux内核这个数据包是一个VXLAN要使用的数据帧，为此，我们需要在内部数据帧前面，加上一个特殊的VXLAN头，而这个VXLAN头里有一个重要的标志叫作VNI，它是VTEP设备识别某个数据帧是不是应该归自己处理的重要标识。在Flannel中，VNI的默认值是1，这也是为何，宿主机上的VTEP设备都叫作flannel.1的原因，这里的“1”，其实就是VNI的值。

然后，Linux内核会把这个数据帧封装进一个UDP包里发出去。

至此，还有一个问题没有解决（其实在UDP中已有解决），就是Linux内核怎么知道UDP包的地址（之前是flanneld进程转发，现在是系统内核）。flannel.1设备实际上要扮演一个“网桥”的角色，在二层网络进行UDP包的转发，在Linux内核里面，“网桥”设备进行转发的依据，来自于一个叫作FDB（Forwarding Database）的转发数据库。那么显然，这个flannel.1“网桥”对应的FDB信息，也是flanneld进程负责维护的。
