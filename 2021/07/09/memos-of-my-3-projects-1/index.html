最近在准备找工作，由于要被问到项目，就走马灯似地回顾了过去一年半的经历。虽然不是什么大的项目，倒也是有所收获；心态上也稍稍发生了一些变化，觉得 CRUD 也不是那么不堪了。这是第一部分，谈谈我的第一份项目。

第一个项目是我做过最久的，大概七个月，也是我收获最多的。这个项目中我负责两个模块，一个是普通的业务模块（CRUD），另一个模块是处于下游的负责大部分生产数据查询的模块，向上游业务模块提供查询接口。

不要犹豫

做这个项目之前，我真的没有自信。从前的我只用 pytorch 搭网络，真正的工程我从没有接触过，因此我也十分感谢实验室能提供这个机会，尽管后面有些许不愉快。
了解到项目基于 Spring boot 框架后，我去找了一些资料，什么 AOP, IoC，看得我一头雾水， 后来我放弃了，直接 github 上找了份入门项目，但我在浏览器中看到一行 Hello, akira！时，我稍稍明白了@controller 的意思。这也是我之后学习任何新技能的方式–先做出 Hello，world，再观察；做出变化，再次观察；等到获得足够多的反馈后，去了解原理。事实上我了解 Spring 的原理是我在写完第一个项目后。

模仿，并改进

在我写了一个小的 demo 后，老师并没有给我太多时间深入学习，于是我直接投入到项目中去，彼时的我还仅仅对 controller、service、mapper 这些层有大致的概念，但是一旦去写，还是我从下手。于是我把项目中其他人写的的模块看了一遍，抄了过来，改了一下，发现可以用，很是兴奋。然后，我把涉及到的注解诸如@Resource、 @RequestParam、 lombok 的官方文档读了一下，尽管不知道原理，但是可以用了。这样的状态持续了几天，我已经可以写稍微复杂的逻辑了。但是我发现了一个问题——我的代码风格及其不规范——由于我模仿别人的代码。比如，我的局部变量用下划线风格表示，我的方法名起的很随便，于是我又花了半天去看源码是怎么命名的，把自己的代码风格确定下来。为此我还去搜索了一下 Google 的命名规范，这也是我之后不管是 go 还是 Python 还是其他语言的命名规范来源。

剪枝以及代码片粒度

如果让我回去看我的第一周写的代码，我多半会拍死我自己。长篇的 If...else，到处重复的代码片。在这之后，我利用continue、 return 去掉了过深的嵌套；用 switch 去掉过多的else ；将重复的代码抽象出功能函数,并参考源码给它们取 doParse， computeIfAbsent，getOrDefault类似风格的名字，并给它们加上了注释；使用 Yoda 风格让我的条件表达式（多半是字符串）避免空指针异常。

使用缓存

差不多半个月左右，我已经能够写出看起来不错的代码了，但是还仅仅停留在能够让人看的水平。我开始思考一下业务逻辑了–我做的业务相当纯粹：定义一个查询模板（表格形式, 日期作为版本号）,列为设备（比如电机），行为指标（电流），能够查出实时数据和统计数据（报表）。对于模板这种不常改变的数据，我就可以把它放到缓存中，一开始我考虑用 HashMap ，后来接触到Redis，决定把它放入Redis中，这样它也可以持久化了。

但是方便是要付出代价的，我很快地面临缓存不一致的问题——也就是明明在我电脑上可以的呀？我需要在模板修改时注意清掉这个缓存，为此，我付出了一个多小时去找代码中的bug。

如果业务写起来麻烦，很有可能是数据库没设计好

当我产生这个感触，已经是我主导（其实是实验室人手不够）第二个项目时了。在写第一个业务时，真的很麻烦去处理各种关系，但是并没有意识到它，一是不懂，二是这些表是老师设计的，老师能犯什么错呢？在这个表中，一个模板关联n个显示项（列,电机），关联m个指标（行,电流），关联，又关联n×m个具体指标（电机的电流）,这几项都是id，此外还有诸个关系表，每次写业务都很麻烦。事实上，只需要让模板关联具体指标就可以了。

尝试些新花样

这个项目中唯一称得上有些难度的要数传输实时数据了。为此，我 Google 了几种方法:

一是前台定时轮询，它的优点是实现起来没有难度，后台接口不用改变，缺点是网络开销大，每次都要三次握手，因此我们第一个排除了它。二是使用长连接，header了加入keep-alive，但是前端仍要轮询，每次传输仍要传输头部，因此也被pass掉了。三是 websocket，它的最大优点是后台能主动推送数据了，而且只需要刚开始连接发生握手，另一个随之而来的优点就是可以有状态查询了，由于每次查询实时数据都要根据查询条件去做逻辑处理获得最终的查询条件，每次根据这个条件去查询最新的数据即可。
为了能够给指定的用户定时发送指定视图的数据，我做了三件工作：

  1. 引入 WebSocket

  2. 引入定时任务线程池，实现动态增删定时任务

  3. 引入守护定时任务，维护失去连接的任务

Github 和 StackOverflow 是好东西

事实上，但我引入 SocketIO 时就出现了一个棘手的问题——项目在本地可以启动，在服务器却无法启动——出现了一个bindException。Google 了一圈，终于在 SocketIO 的 Github 某个 issue 中看到了一样的问题，居然是 socketio 的配置错误，服务器和本地使用的都是我主机的地址，它们使用的配置都是application-dev.yml这一份文件。之后，我把本地开发和服务器的配置分开在两个文件来写，至此，我也了解了 Spring boot 中 profile 的使用。

懂些前端

当我第一次写返回分页的数据时，我没有返回记录的 id，因为我任务它是没有必要的，它都没有在页面中展示，但事实上，我写的删除数据的接口接收的 id 正是从这里来的，前端用这个 id 作为 vue-for 中的id。这是无知的代价。另一个问题是，将问题过度复杂化，比如一个接口接收的数据是字符串类型的日期，我取在这里写了大量无用的判断代码，而实际上这个日期不是用户通过键盘输入的，而是用日期选择器选的。

注意设计模式

在我完成第一个模块后，由于完成的还不错，我又被委派了第二个任务。这个任务要求能够根据数据指标动态地去不同数据库，不同数据表中查询（采样）数据，采样的方式为给出一个时间区间，每隔几个数据点，或每隔一个时间间隔，查询出数据点上的数据或小间隔的统计值；数据也应该考虑是否为累计值（比如电表），此外数据也有量程（即某个时间点的数据可能比之前的小）；之后，查询出来的结果集有可能不是某个业务想要的形式，需要适配一下。 显然，这个任务更注重扩展性和对查询场景的抽象，因此我将这个任务写成了一个框架，利用模板方法来抽象查询的行为（查询条件的封装，查询过程，查询结果集的变换）；利用工厂模式来获取不同查询器（mapper，不如MySql Mapper，Oracle Mapper，SQLServer Mapper）的单例；利用策略模式来分解采样策略（隔点采样，隔时采样）；利用适配器模式来变换结果，并且利用多线程机制来加速变换：

查询优化

这个任务的难度一是对查询过程的抽象建模；二是查询的优化。查询优化我个人总结了三条：1. 减少查询次数 2. 优化 SQL 语句 3. 减少查询结果集大小。

项目中需要由指标ID获取指标数据源所在数据库，所在数据表，数据源优先级，而这个信息一旦确立是很难改变的，因此需要将它放入缓存中，否则每次查询数据都要去数据库中查一遍，极大浪费时间。

优化 SQL 语句 1. 依靠建立索引，以此我在这里对采样时间建立了索引，2. 减少搜索范围，利用 Oracle 的 with ... as ()机制，先缩小查询范围，再在这个范围中找 3. 变换查询条件，比如查询某天的数据，可以变换为查询两个时刻间的数据。

项目中涉及到每隔一段时间查出一个数，已经累计数据，需要后一个数据减去前一个数据，如果小于0，需要加上量程。如果这段逻辑要在数据库外处理，数据的传输都是个大麻烦，比如查询一年的数据，每分钟一条，一年就是50多万条。当时翻遍了 stackOverflow，终于找到这么一个解决方案：

没有什么比沟通更重要的了

写到最后，我才写沟通，不是我觉得它不重要，事实上它是最重要的，我认为它是一个工程师应具备的素养，决定一个项目的死活。这个项目中我深感沟通是门艺术，会沟通的人往往能把握沟通的时机，沟通的方式，沟通的态度。这个项目中与我合作的大都是学生，没有经历过社会的历练，有些加了微信也不说句问候的话，自己我要什么什么接口；有些在晚上十一二点找我联调，没把我气的半死。到了第三个项目，与我接触的是个工作过的人，他不会在我休息的时候打扰我，每次找我都会问句在吗，你好之类的，遇到不懂的也会谦虚请教，虽然每次都会带来新需求，但是不会让我感到厌烦，合作非常愉快。沟通采用的媒介也非常重要，我们采用线上开会时，一个问题要讲很久才能讲通，对于我这样表达能力差的真的很痛苦；而线下，我们可以在黑板上写写划划，沟通的效率大大提高。

我真的懂 crud 吗

做了几个项目后，我发现我厌倦了这些无聊的业务，无法就是增删查改，再复杂能复杂到哪去。然而，当我去学习 Restful 时，我再看我们的项目，连 endpoint 的命名都不是 restful 风格，通常都是以getSomething的形式呈现，项目中也只有GET和POST方法，这很难让人相信我真的会写“业务”。

其次，我们用的 mvc 模型，controller，service，dao层往往很难区分，我的一套逻辑是，controller层做一些简单的校验，将核心的业务分发给service层，dao负责与数据库的交互；然而我看别人的代码controller层做了大量业务，这很难说是这个人的问题，因为我们的架构就摆在这儿——它允许我们这么做。service调用的获取数据接口是由dao层定义的，显然这又是另一个逻辑问题，难道不应该在service层定义，由dao实现吗？

直到最近，我才接触到DDD，领域驱动开发，一下子就解决了我的困扰。CRUD，其实也是有学问的。
