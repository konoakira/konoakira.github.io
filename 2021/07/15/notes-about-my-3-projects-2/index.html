这一篇回顾我做的另外两个项目，感触并没有上篇那么多，更多的是技术上的处理与思考。

第二个项目

第二个项目是一个微信公众号引流项目，如下图，用户登录微信小程序获取打印内容二维码，在打印终端上扫码打印，打印时会播放广告。再从打印的票上获取要关注的公众号，公众号会发送链接，点击回到小程序领取红包。

这个项目设计到四方，服务器端，小程序端，Web 管理端，打印终端。值得夸耀的是，服务器端和Web 管理端的代码完全是我本人写的；小程序端我的页面是我画的，登录逻辑是我实现的；打印终端 C# 授权登录逻辑也是我实现的。这个业务的流程也是由我本人设计并验证的，虽然业务很简单，但是还是有值得推敲的地方的。

可行性分析

不得不说，可行性分析是业务的第一位的，应该排在需求的前面。我们面临的第一个问题就是能否发红包，该怎么发红包，在哪里发红包。根据腾讯公众号平台的文档（不得不吐槽做的太差了，该踩的坑一样不会少踩），商户需要注册90天，且有近30天的流水。这一条我们就不满足，索性客户有其他的商户号，又是疫情期间，可以临时顶上拖一阵。第二个问题就是怎么发？我们希望在小程序中发，调用小程序的 API，可是气人的来了，我们需要用户在公众号下的openid，是的，小程序下的openid和它不是一个，也就是说我们不能用这个接口！还好，我发现腾讯有个公众号平台，商户号（开通企业到个人付款）绑定它，小程序再绑定它，皆可以调用企业到个人付款的API，并且是使用小程序的openid。我现在说起来简单，可当时真是折磨的不行。（顺便一提，这个微信红包不能少于三毛钱，有一个坑）。

架构的安全

这是我第一次认真思考这个问题，之所以会思考它，是因为我们的业务涉及到金钱。

传输安全

这没什么好商量的，只能 HTTPS 传输，防止窃听，篡改和假冒风险。事实上，腾讯支付要求必须要 HTTPS 传输（证书由商户下载给我们），并且对请求参数进行签名。此外，小程序也要求 HTTPS 传输。

授权安全

我们要保证打印终端不会被盗用，被假冒。（除了 HTTPS 传输）每台打印机在首次使用时会被要求输入一个唯一的32位授权码，它也可以理解为数据表中的一个键，这样可以保证1. 设备一定是提前存在与数据库中 2. 激活设备时会查询设备的状态，这保证了其一定不会被二次激活。

我们又是怎么知道打印时机器的编号呢？这里就要说我们的 Token 机制了，传输时服务器生成 JWT token 交给终端，以后终端每次传输携带它。那么是怎么获取 Token 的呢？终端授权成功会，会把终端 id 利用对称加密算法得到的字符串写入注册表中，登录服务器时带上这个加密字符串，服务器解密验证发放 Token。

交易安全

首先，交易不是一个幂等的操作，不可能用户点一次领取红包就发一次。这一点其实腾讯已经想到了，我们的系统再不安全，一天也只能给一个用户发最多50块钱的红包。其次，腾讯支付要求交易时携带交易号，而这个交易号是用商户号+日期+10位一天内不能重复的数字构成，我们生成交易号后存入数据库，这一层也保证了安全。最后，腾讯要求只能由白名单中的 IP 才能调用发放红包的接口。

我们的工作似乎就变成了保证订单号唯一，不会多发红包即可。这一点接下来会详谈。

数据库安全

这里真的就很简单了，有时我调试时都会觉得麻烦。首先是强口令。其次是禁止远程访问，最后开启防火墙。这里我想起我们上线就遇到的 bug，开发已经调试时我们所有人的微信昵称没有一个有 emoji 的，上线时好多人都有 emoji，导致无法存入数据库。后来，我把编码改为utf8mb4解决了这个问题，我是怎么发现的，因为我让客户去给我拍下出问题的微信号，它们的共同点就是都是 emoji。

日志的重要性

我之前很少记录日志，可能是我对自己的自信，能出现什么 bug 呢？可现实就是我们无法预料到所有场景，比如 emoji 问题，比如红包不能少于三毛钱，比如前端传输时加了引号，比如商户号没有余额等等，由于加了 logging，我能很快定位这些问题。

如何保证不会超卖

这里会出现超卖问题吗？老实说，概率很小。解释这个问题，还要先说明我是何时从广告主处扣钱。并非每次打印都会扣钱，打印终端发送打印请求获取打印内容及广告内容，这一部并不会扣钱，因为打印有可能失败（比如打印纸张不够）。打印成功后，终端会发送回执，这一步会扣钱。而打印就注定是同步过程，我们有五个打印机，事实上同时发生打印成功的概率还是很小的。尽管如此我还是加了个 synchronized，并发量最大也就是5，性能没问题的。

订单号生成策略

订单号何时生成？

显然不能是从广告主处扣钱后，因为大多数人很有可能不会去领钱，因此订单号要在其领红包时生成。

订单号怎么生成？

前面讲过，订单号由商户号+日期+一天能不能重复的10位数字。我的第一反应是这个数字可以查数据库中的已领取的票数决定，显然这是不对的，可能会有几个人同时领。后来，我回忆起 Redis 有个自增功能，由于它是同步的，于是这个数字就有 Redis 生成。

订单失效机制

生成票后，三十分钟，如果用户没有领红包，就让它失效，并把钱还给商家。一开始我想用延时队列做这件事，单开一个线程一直去队列中拿票，后来觉得不够优雅，作罢。因为我想起来 Redis 有个过期键机制，每次生成一个票，就在 Redis中插入一个 30 分钟的键，然后监听过期事件。注意，这里我还考虑了一个懒检测机制，如果某次没有收到监听时间，用户领红包时如果状态是未领取也会再判断一下是否过期（其实这一步是获取用户红包金额时一起做的）。

容错机制

发钱时由于通过网络请求腾讯服务器，那它可能就出现网络问题。这里我加了一个 FailOver 机制，如果出现异常，那么就重试再发一次。可能还会失败，怎么办呢？这里就要提我设计的状态机制了。一张票的状态可以是未领取（0），已领取（1），已失效（2），我又加了个领取中（3，4，5）。发红包时把状态设为领取中（3），发红包失败返回让用户重试（多动一下拇指而已），大多数人都会再按一下。如果顾客没有重试，那么在上一步的订单失效机制触发时，如果这个状态还是3，我就再给这个键续30分钟，同时把状态改为5；如果这个状态是5，那我就认为用户不愿再领了，直接把钱还给商家。

事件机制

其实，可以看到很多地方都会涉及到商家的状态（如果它账户下没有余额，或者充值后，或者红包退还都会改变状态），这里我就引入了刷新商家的事件。每次其余额更改时都会触发该事件，这样我就不要重复写更改状态的逻辑了。

终端健康

这里就用到的简单的心跳机制，我也在这里加入了容错机制。由于终端是连接地铁里的 Wi-Fi，信号难免会差，很可能某次心态没发过来。我在这里容忍三次失联，每次收到心跳时与数据库最后更新时间比，如果超过三次心跳间隔就通过 WebSocket 向 Web 端发出警告。

扫码枪编码

其实这不算什么大问题，由于扫码枪扫二维码得到的是一个定长的字符串，这就要求我们的编码具有可扩展性。我们把前32位设为用户 id，把33-34 设为打印内容大类（单词，星座），35-36 设为打印内容小类（天蝎，雅思），这样的设计就满足了我们的需求。日后我们也再增加打印内容（总共100类，远远足够），而不必更改终端代码。

第三个项目

第三个项目是一个将任意可执行代码（这里叫算法资源）封装发布成外界可请求的 url 形式，对就是云函数（后者叫 FaaS）这一套。但是，做的及其痛苦，我认为它已经失败了，尽管现在还在做。失败的原因有几个：

想的太多

我们的老师同大多数不懂技术的创业者一样，想的太多。我们实验室是个很小的实验室，本来就不多人，结果我们核心功能都没做出来，他就在畅想怎么获利，会员制，计算资源计费…..只有两个人研究核心功能，其他人做社区功能或者其他项目，excuse me？

软件工程也是一门管理艺术

凡是设计到工程，必然设计到分工，人员的调度，对工作难度的估计，对团队成员的能力的评估。我们用到的技术不再仅仅是 Spring boot 那一套了，Kubernetes，Docker，Linux 都要有人懂得。是的，我们每一个人懂得。现在的状态就是一个不懂的人带着几个不懂的人瞎搞。

公平的制度

这一点我在做第一个项目时就感受到了，本来第二个任务是分配给两个人做的，结果落在了我一个人身上，用他们的话术就是能力越大责任越大。可我只感受到了责任。第二个项目是我在开题时做的，大部分工作都是我一个人完成，包括担任客服角色。做第三个项目我正面临找工作……当然，我也懂得感恩，懂得感谢实验室的资源，可即使如此，即使抗压能力再强，也受不了这样的剥削。

尽管如此，我还是学习了 Docker 技术，k8s 的搭建，控制器模式，和一些基础的 API object，也算是受虐狂的意外收获吧。
